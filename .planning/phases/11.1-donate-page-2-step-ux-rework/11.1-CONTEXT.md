<decisions>
## Implementation Decisions

### UX Flow — 2-step method-then-amount

- **Step 1 (landing state)**: Render two large method-picker cards side-by-side (equal width at sm+, stacked at xs). Each card is a single clickable surface — no chips, no input, no inner CTA. Clicking the whole card advances to Step 2 for that method.
  - **Fiat card**: Iconify `solar:card-bold-duotone` in navy squircle tile + title "Carte bancaire" / "Credit card" + subtitle "Visa, Mastercard, Apple Pay…"
  - **Bitcoin card**: Iconify `bitcoin-icons:bitcoin-circle-filled` in navy squircle tile + title "Bitcoin" + subtitle "On-chain & Lightning" with "⚡ Lightning supported" tag
- **Step 2 (amount state)**: Show a SINGLE centered card containing:
  - A small "← Changer de méthode" / "← Change method" link at top-left of the card (navigates back to Step 1)
  - The chosen method's icon tile + label at the top (smaller than Step 1, acts as a header)
  - Fixed 4-column amount chip grid at sm+ (5/10/25/50 €), 2x2 at xs
  - The custom-amount text input paired with right-edge stacked chevron stepper (preserved from Phase 11)
  - One full-width "Soutenir" / "Support" CTA at the bottom that IS the sole redirect trigger

- **State transitions**: Use a local `useState<"idle" | "fiat" | "bitcoin">("idle")` hook inside `DonateCards.tsx`. No URL param, no route change, no full remount — it's a single client component with conditional rendering. "← Change method" resets state to `"idle"` and clears any selected amount.

- **Amount selection semantics (CRITICAL — changes Phase 11 behavior)**:
  - Clicking an amount chip NO LONGER redirects. It only sets local `selectedAmount` state and visually marks the chip as selected (accent translucent fill + accent border + glow soft — the spec from Phase 11's chip states that was never wired up).
  - Typing in the custom input also sets `selectedAmount`. Stepper chevrons bump by ±1 EUR bounded by MIN_AMOUNT=1 / MAX_AMOUNT=999.
  - The CTA button is the SOLE redirect trigger:
    - Fiat path → The CTA is an `<a>` tag with `href={stripeLinks[amount] || stripeLinks.custom}` and `target="_self"`. If `selectedAmount` is a chip value (5/10/25/50), use that Payment Link directly. Otherwise use `stripeLinks.custom` (Stripe's custom-amount Payment Link).
    - Bitcoin path → The CTA is a `<button type="submit">` inside a `<form>` that POSTs to `${btcpayConfig.serverUrl}/api/v1/invoices` with `storeId`, `orderId`, `checkoutDesc`, and `price=${selectedAmount}` in form-data (preserves Phase 11's working flow).
  - **Disabled CTA state**: When no amount is selected yet (no chip picked, no custom digits entered), the CTA is `disabled` (muted visual) and does nothing on click. This disambiguates the interaction — user must first pick amount, then confirm.

### Color — CTA gradient swap

- Replace the navy CTA gradient `linear-gradient(135deg, #1A4E8A, #0F3460)` with the brand **Accent Blue** gradient: `linear-gradient(135deg, #3D7EFF, #2563EB)`.
- Apply a soft glow on hover: `box-shadow: 0 8px 24px rgba(61,126,255,0.35)`.
- The disabled state uses a muted surface (`backgroundColor: rgba(255,255,255,0.06)`, `color: rgba(255,255,255,0.35)`, `cursor: not-allowed`, no gradient).
- This aligns with the Nav pill button (which already uses `#3D7EFF` per brand kit).
- The Step-1 method cards keep their current navy-surface aesthetic (Surface `#161C2C` + Border `rgba(255,255,255,0.07)`). Only the CTA changes.

### Hover / focus affordance — Step 1 cards

- Each method card gets a clear interactive affordance: on hover, scale `1.02`, border brightens to `rgba(255,255,255,0.14)` (Border Hi), add glow `rgba(61,126,255,0.12)` (Glow Soft).
- Keyboard: each card is a `<button>` (not `<div>`) with `aria-label="Soutenir par carte bancaire"` / `"Soutenir en Bitcoin"` so tab order + Enter/Space activation work.

### i18n keys (additions + renames)

New keys under `Donate` namespace in both `fr.json` and `en.json`:
- `method_fiat_title` → "Carte bancaire" / "Credit card"
- `method_fiat_subtitle` → "Visa, Mastercard, Apple Pay…" / "Visa, Mastercard, Apple Pay…"
- `method_bitcoin_title` → "Bitcoin" / "Bitcoin"
- `method_bitcoin_subtitle` → "On-chain & Lightning" / "On-chain & Lightning"
- `method_bitcoin_lightning_tag` → "⚡ Lightning supporté" / "⚡ Lightning supported"
- `change_method` → "Changer de méthode" / "Change method"
- `amount_prompt` → "Choisissez un montant (€)" / "Pick an amount (€)"
- `custom_amount_label` → "Montant personnalisé" / "Custom amount"
- `cta_support` → "Soutenir" / "Support" (reuses existing `custom_submit` value; add as alias for clarity in the new flow — OR reuse `custom_submit` directly, planner's discretion)

Keep (unchanged): `meta_title`, `heading`, `motivation`, `thank_you`, `why_support_*` (keep the Why support? bullets section below the card — they still add value).

Remove (deprecate in JSON — can be left in place as dead keys without breaking anything, OR removed if the planner wants a clean sweep):
- `chip_label_fiat` and `chip_label_bitcoin` from Phase 11 were used for per-chip aria-labels; with the new flow they may or may not still be needed depending on the planner's aria-labelling choice. Planner's discretion: reuse them if aria-labelling chips makes sense, otherwise delete.

### "Why support?" section — keep

The inline-bullets "Pourquoi soutenir ? / Why support?" section shipped in Phase 11 (plan 11-05) stays exactly as-is. It appears below the card in BOTH Step 1 and Step 2 states (page-level, outside DonateCards component). Zero changes to that block.

### Mobile

Same 2-step flow. In Step 1, the two method cards stack vertically. In Step 2, chips drop to 2x2. Nothing else changes.

### Accessibility

- Step 1 method cards: `<button>` elements with descriptive `aria-label`.
- Step 2 back link: `<button type="button">` with visible text "← Changer de méthode" + `aria-label="Retour au choix de méthode"`.
- Amount chips: `<button type="button" aria-pressed={selected}>`.
- Custom input keeps `inputMode="numeric"`, `pattern="[0-9]*"`, `aria-label` set via i18n.
- Disabled CTA: `aria-disabled="true"` in addition to native `disabled`.

</decisions>

<specifics>
## Specific Ideas

- **Transition animation (Claude's discretion, recommended but optional)**: Use a CSS `transition: opacity 200ms ease, transform 200ms ease` cross-fade when switching Step 1 → Step 2 and back. Keep it minimal — no framer-motion dependency. If it adds complexity, ship without animation (snap transition is acceptable).
- **Brand icon consistency**: Reuse the EXACT same Iconify icon names from Phase 11 (`solar:card-bold-duotone`, `bitcoin-icons:bitcoin-circle-filled`). No new icons.
- **Navy squircle tile**: Reuse the existing tile styling (`bg-[linear-gradient(135deg,#0D2040,#071020)]`, `rounded-[18px]`, `w-12 h-12`). On Step 1 it's rendered larger (e.g., `w-16 h-16`) to give the method cards more visual weight; on Step 2 it returns to `w-12 h-12` as a header badge.
- **Layout constraint Step 2**: The single centered card has `max-w-[480px]` so it doesn't feel stranded on a wide viewport. Step 1 grid is `max-w-3xl`.

</specifics>

<deferred>
## Deferred Ideas

- **Loading state during redirect**: If a user clicks the CTA and the Stripe/BTCPay page takes >500ms to load, we could show a spinner on the button. Not in scope for 11.1 — would require measuring real-world redirect latency first.
- **Analytics events**: Tracking `method_selected` / `amount_selected` / `cta_clicked` events. Out of scope — no analytics infra yet.
- **Remember last choice**: Persist the user's preferred method in `localStorage` to skip Step 1 on return visits. Out of scope — adds privacy-policy surface area that contradicts the "no tracking" ethos.

</deferred>
